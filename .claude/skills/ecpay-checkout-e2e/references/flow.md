# 流程細節參考(實測記錄)

這份文件記錄實際跑過一次完整流程後觀察到的頁面結構與互動細節。ref 編號(如 `e22`)只是某一次執行的快照結果,每次重新 navigate 都會變,**不要直接套用編號**,只把這份文件當作「畫面長什麼樣、該找什麼角色/文字」的對照。

## 1. 登入頁 (`/login`)

- 未登入訪問 `/` 會自動 302 轉址到 `/login`。
- 頁面同時有兩個文字是「登入」的按鈕:
  - 分頁籤按鈕(`登入` / `註冊` 切換,在表單上方)
  - 表單送出按鈕(在 email/密碼欄位下方)
- 用 `getByRole('textbox', { name: 'you@example.com' })` 抓 email 欄位,`getByRole('textbox', { name: '請輸入密碼' })` 抓密碼欄位最穩定(用 placeholder 當 accessible name)。
- 送出按鈕要用 `page.locator('form').getByRole('button', { name: '登入' })` 精準鎖定表單內按鈕,避免點到分頁籤。
- 成功登入後 URL 變成 `/`,Title 變成「首頁 - 花漾生活」,導覽列會多出「我的訂單」連結與右側「後台管理 / Admin / 登出」(代表是 admin 帳號)。

## 2. 首頁 (`/`)

首頁有兩個商品卡片區塊,結構不同:

- **「今日鮮花精選」**(輪播/精選區):卡片只有圖片、標題、價格,**沒有「加入購物車」按鈕**。
- **「探索所有花藝」**(全部商品區):卡片才有「加入購物車」按鈕。這裡才是要點擊的地方。

商品範例(價格會隨後台資料調整,不要寫死比對):粉色玫瑰花束 NT$1,680、白色百合花禮盒 NT$1,280、繽紛向日葵花束 NT$980、紫色鬱金香盆栽 NT$750、乾燥花藝術花圈 NT$1,450、迷你多肉組合盆 NT$580、經典紅玫瑰花束 NT$3,980、季節鮮花訂閱(月配)NT$890。

點擊加入購物車後頁面不會跳轉,需要自行導向 `/cart` 確認結果(或檢查導覽列徽章數字變化)。

## 3. 購物車頁 (`/cart`)

- 滿額免運會顯示綠色提示「✓ 已達免運門檻!」。
- 每個品項有 `-`/`+` 數量按鈕與垃圾桶圖示刪除按鈕(🗑)。
- 下方摘要區塊:商品小計 / 運費 / 總計,「前往結帳」按鈕在最下方。

## 4. 結帳頁 (`/checkout`)

- 三個必填欄位:收件人姓名、Email、收件地址(都是純文字輸入框,placeholder 分別是「請輸入收件人姓名」「請輸入 Email」「請輸入收件地址」)。
- 右側「訂單摘要」會列出品項 x 數量、小計、運費、總計。
- 送出按鈕文字是「確認送出訂單」,點擊後若成功會直接整頁導向 ECPay 網域(不是用彈窗或 AJAX,是真的 302/表單 POST 過去)。

## 6. 「將跳轉至銀行頁面」提醒彈窗(關鍵陷阱)

點擊「前往付款」後,畫面上方按鈕區域看起來像被替換成 `active` 狀態的連結,同時畫面**下方**會浮現一個 `role="modal"` 的彈窗(class 含 `simplert`),內容是:

> 綠界科技貼心提醒您,將跳轉至銀行頁面進行後續付款!因網路ATM屬於即時交易…

彈窗有一個「關閉」按鈕。**這個 modal 會攔截整個頁面的點擊事件**,如果在它還顯示時嘗試再點一次「前往付款」,Playwright 會不斷重試最後 timeout(實測對應錯誤訊息:`<div role="modal" ...class="simplert simplert--shown">intercepts pointer events`)。

正確做法:點擊「關閉」按鈕關掉提醒視窗,瀏覽器會自動完成原本要做的跳轉,不需要再點第二次「前往付款」。

## 7. 模擬銀行頁面(Mock 環境專屬)

關閉提醒視窗後會自動跳轉到類似 `https://pay-stage.ecpay.com.tw/MockMPPost/LandWebAtm` 的頁面(Title 直接顯示銀行+ATM 的英文代號,例如 `LandWebAtm`)。

這個頁面是 ECPay **測試站(staging)專屬**的模擬器,本質是一個顯示交易參數的表單,常見欄位:

- `RC`:回應代碼,`0` 代表交易成功
- `MSG`:文字訊息,例如「交易成功」
- `CurAmt`:交易金額(應與訂單金額一致)
- `TrnDt` / `TrnTime`:交易日期/時間

頁面下方有一顆「Save」按鈕,點擊後等同於模擬「使用者在真實網路ATM操作完成」,會把這組模擬結果送回 ECPay。**正式環境不會有這個頁面**——如果之後串接正式環境(`ECPAY_ENV` 不是 staging),這一步會被真實銀行頁面取代,需要另外處理(超出本 skill 範圍,屆時應停下並告知使用者環境已變更)。

## 8. ECPay 付款成功頁

URL: `https://payment-stage.ecpay.com.tw/bank/PaymentCenter/cntnotlogin/webatm/result`,Title:「付款成功|綠界科技」。

顯示訂單編號、商店名稱、付款方式(網路ATM)、商品明細、應付金額,以及一個「返回商店」連結,連結目標是 `http://localhost:3001/orders/<order-uuid>?payment=pending`(注意 query string 是 `pending`,這是導回時的初始狀態,不代表後端訂單真的還沒處理完)。

## 9. 商店訂單詳情頁 (`/orders/<id>`)

- 成功處理完 ECPay callback 後,頁面會顯示綠色提示文字「付款成功!感謝您的購買。」與狀態標籤「已付款」。
- 若剛跳轉回來時還沒看到「已付款」(可能後端 callback 還沒處理完),**重新整理頁面**而不是直接判定失敗。
- 頁面同時列出收件資訊、訂單資訊(訂購日期)、商品明細表(商品名稱/單價/數量/小計)與訂單總計,可以用這些資料跟結帳頁/購物車頁的金額交叉核對,確認整個交易鏈路(建立訂單→ECPay 付款→callback 更新狀態)是一致的。
