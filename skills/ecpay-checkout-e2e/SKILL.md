# ECPay 結帳 E2E 測試(花漾生活 backend-project)

## 這個 skill 做什麼

驅動 Playwright MCP 在本機瀏覽器中,完整跑一次「啟動伺服器 → 登入 → 加入購物車 → 結帳 → 綠界(ECPay)金流付款 → 確認訂單已付款」的流程,並回報每一步的實際畫面狀態給使用者。本專案串接的是 ECPay **測試站(staging)**,所有付款都是模擬交易,不會產生真實金流,因此可以放心反覆執行。

跑這個流程之前,先讀一次 [references/flow.md](references/flow.md) —— 裡面記錄了每個頁面的實際 DOM 結構、已知的互動陷阱(例如 ECPay 的提醒彈窗會擋住按鈕),這些是直接跑過真實流程才知道的細節,光看程式碼猜不出來。

## 預設參數(可被使用者要求覆寫)

| 項目 | 預設值 | 何時覆寫 |
|---|---|---|
| 網址 | `http://localhost:3001` | 使用者指定其他 port 或部署網址時 |
| 帳號 | `admin@hexschool.com` | 使用者要求用其他帳號(例如測試一般會員權限差異)時 |
| 密碼 | `12345678` | 同上 |
| 付款方式 | 網路ATM(WebATM) | 使用者要求測信用卡、超商代碼等其他方式時 |
| 銀行 | 臺灣土地銀行 | 使用者指定其他銀行時 |
| 商品 | 商品列表第一個有「加入購物車」按鈕的品項 | 使用者指定特定商品名稱時 |

沒有特別說明時,直接採用上述預設值執行,不需要每次都反問使用者。如果使用者的描述偏離預設(例如要測信用卡而非ATM),按使用者說的調整,其餘步驟維持不變。

## 執行步骤

### 1. 確認伺服器是否已啟動

```bash
netstat -ano | grep ":3001" | grep LISTENING
```

- 若已有 process 在監聽,直接使用現有伺服器,不要重複啟動(會 port 衝突)。
- 若沒有,在背景啟動:`npm run start`(注意:這個指令會先跑 `css:build` 再啟動 server,需要幾秒鐘,啟動後用工具讀取背景輸出確認看到 `Server running on port 3001` 才算就緒,不要憑空假設已啟動)。
- 若埠號被其他不相關的程式佔用導致啟動失敗,回報給使用者,不要自行砍掉佔用該 port 的行程。

### 2. 開啟首頁並登入

導向 `http://localhost:3001`。本專案未登入時會自動轉址到 `/login`。

用 snapshot 找到 email/密碼欄位與「登入」按鈕(頁面上同時有分頁籤的「登入」按鈕與表單內的「登入」送出按鈕,兩者文字相同,送出時要鎖定表單內那顆,不要鎖到分頁籤按鈕)。填入帳密後點擊送出,確認頁面轉址回首頁(`/`)且 Page Title 變成「首頁 - 花漾生活」。

若登入失敗(留在 `/login` 或出現錯誤訊息),截圖/snapshot 給使用者看,停止後續步驟並回報,不要重試超過一次。

### 3. 加入商品到購物車

首頁「探索所有花藝」區塊中的商品卡片才有「加入購物車」按鈕(上方「今日鮮花精選」輪播區塊的卡片沒有這顆按鈕,不要點錯區塊)。點擊任一商品的「加入購物車」。

導向 `/cart` 確認:
- 購物車內出現剛加入的商品
- 導覽列「購物車」連結旁出現數量徽章(如 `購物車 1`)

### 4. 結帳

點擊購物車頁的「前往結帳」,進入 `/checkout`。填寫收件人姓名、Email、收件地址(隨意填寫合理測試資料即可,這是內部訂單資料不是金流資料)。點擊「確認送出訂單」。

成功的話頁面會跳轉到 `https://payment-stage.ecpay.com.tw/...`,Page Title 為「選擇支付方式|綠界科技」。這個跳轉證實了後端訂單建立 + ECPay 訂單建立(`db.transaction()` 那段:建立訂單→寫入品項快照→扣庫存→清空購物車)都成功了 —— 如果沒跳轉,通常是表單欄位驗證沒過或後端 500,回報錯誤訊息給使用者,不要硬點下一步。

### 5. 在 ECPay 選擇付款方式與銀行

ECPay 頁面付款方式是一排 `listitem`(Credit / ApplePay / TWQR / WeiXin / iPassPay / JKOPay / **WebATM** / ATM / BarCode / CVS / GWPay / BNPL)。「ATM 網路交易」對應的是 `WebATM`(網路ATM),不是 `ATM`(那個是 ATM 虛擬帳號,流程不同,沒有選銀行這一步,而是給一組轉帳帳號)。選錯會導致後面找不到銀行下拉選單。

點擊「網路ATM」listitem 後,畫面會換成銀行選擇表單,用 `combobox` 選擇銀行(預設「台灣土地銀行」)。

### 6. 前往付款並完成模擬交易

點擊「前往付款」連結後,ECPay 會彈出一個提醒視窗(class 為 `simplert`),告知「將跳轉至銀行頁面進行後續付款」。**這個彈窗會用 modal 蓋住整個畫面並攔截點擊事件**,如果不先點「關閉」按鈕直接點第二次「前往付款」會一路 timeout —— 先點「關閉」,彈窗消失後頁面才會自動完成跳轉(不需要再點一次前往付款)。

跳轉後會到 ECPay 測試站模擬的銀行頁面(網址類似 `pay-stage.ecpay.com.tw/MockMPPost/...`),這是一個顯示交易參數的表單(RC=0 代表會成功、MSG 顯示「交易成功」),點擊頁面上的「Save」按鈕送出,等同於模擬使用者在真實網路ATM完成轉帳。

送出後會回到 ECPay 的「付款成功」頁面,確認上面顯示的訂單編號、付款方式(網路ATM)、金額與步驟 4 建立的訂單一致。最後點擊「返回商店」連結。

### 7. 確認訂單狀態

返回商店後會落在 `/orders/<id>?payment=pending` 訂單詳情頁。**第一次看到頁面時付款狀態可能還顯示處理中**——ECPay 的付款結果是透過後端 callback 非同步更新訂單狀態的,不是前端跳轉那一刻就同步寫入。如果畫面還沒顯示「已付款」,等待數秒後重新整理頁面再檢查一次,而不是立刻判定失敗。

最終確認畫面包含:
- 「付款成功!感謝您的購買。」提示文字
- 訂單狀態標籤為「已付款」
- 商品明細與總金額與購物車/結帳頁一致

把這些確認結果整理成簡短摘要回報給使用者(訂單編號、商品、金額、付款方式、最終狀態),不需要逐步重複過程細節。

## 失敗情境處理原則

這類瀏覽器自動化流程最常見的失敗不是「步驟錯了」,而是「畫面狀態還沒到」或「有東西擋住了預期的元素」。遇到點擊失敗、找不到元素、或頁面內容跟預期不同時:

1. 先用 snapshot 重新看一次當下實際畫面,不要憑記憶中的 ref 編號硬點(每次 navigate 或互動後 ref 都可能變動)。
2. 檢查是否有彈窗/modal 蓋住了目標元素(ECPay 金流頁面常見),需要先關閉。
3. 若同一步驟重試 2 次仍失敗,停下來把目前畫面狀態回報給使用者,說明卡在哪一步、看到了什麼,讓使用者決定要不要調整,而不是無限重試或跳過該步驟硬講「完成」。

## 何時不需要走完整流程

如果使用者只想驗證部分功能(例如「測一下登入會不會壞」「加入購物車的數量徽章會不會更新」),只執行對應的步驟,完成後即可回報,不必勉強跑到付款完成。
